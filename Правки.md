# Правки / техдолг (чтобы не забыть)

## 1) Яркий баг: block3 (Яндекс‑отзывы) падает из‑за несовпадения Chrome ↔ ChromeDriver

### Симптомы (логи worker)
- `SessionNotCreatedException: cannot connect to chrome ...`
- `ChromeDriver only supports Chrome version 145`
- `Current browser version is 147.x`

### Почему так
В Docker‑образе **Chrome обновился**, а **ChromeDriver/undetected_chromedriver** остался на **145** → парсер `parse_yandex_reviews.py` не может поднять сессию.

### Варианты исправления
- **Вариант A (быстро, “железно”)**: зафиксировать установку **Google Chrome на конкретной версии** в `backend/ms-v2/Dockerfile` (и не давать ему “плыть” при rebuild).
- **Вариант B (правильнее)**: доработать `parse_yandex_reviews.py`/инициализацию `undetected_chromedriver`, чтобы **автоматически подтягивался драйвер под текущую версию Chrome** (или явно передавать `--chrome-version`).
- **В репозитории**: в `parse_yandex_reviews.py` по умолчанию вызывается **`_detect_chrome_major_version()`** (как в `block5_tech`); явный `--chrome-version N` переопределяет.

### Ожидаемое поведение после фикса
- block3 перестаёт падать на старте сессии
- если отзывы всё равно не собрались по другим причинам — пайплайн может завершиться как `done_partial`, но **не из‑за mismatch версий**

## 5) Логи воркера: `ingest failed: 401` и block3 (Chrome) — разные проблемы

- **`ingest failed: 401`** в `analysis_runs` / ответе poll — это **не воркер**. Исправляется на стороне **Edge Functions**: в `ms-v2-poll` для вызова `ingest` нужны корректные заголовки (в т.ч. `apikey` как anon public + `Authorization` с service role), в `ingest` — явное определение пользователя при JWT с ролью `service_role` из `user_id` в теле. После правок в репозитории — **задеплоить** `ms-v2-poll` и `ingest`.

- **ChromeDriver 145 vs Chrome 147** в **block3_reviews** (`parse_yandex_reviews`) — **отдельная тема** от ingest: в Docker-образе нужно **выровнять версии Chrome и ChromeDriver** (или зафиксировать установку Chrome), либо задать **`--chrome-version 147`** в скрипте, если он это поддерживает. Подробнее см. **§1** выше.

## 6) Аудит “чистки репозитория” (что лишнее/что оставить) — 2026‑04‑21

### Что НЕ мусор (оставить как есть)
- **`marketscoup/`**: используется и документирован; есть проверки в CI.
  - Упоминания: `README.md` (CLI), `.github/workflows/ci.yml` (python smoke: импорт + `marketscoup.cli --help`).
- **Корневые `package.json` + `package-lock.json` + `vercel.json`**: нужны для деплоя фронта на Vercel из монорепы.
  - Идея: чтобы Vercel выбирал **Node/Vite** (workspace `mark/MarketScope`), а не **Python** из `pyproject.toml`.
- **`render.yaml`**: blueprint для Render (ms-v2-api + ms-v2-worker + Redis); файл реально описывает прод-сервисы.
- **`requirements.txt` + `pyproject.toml`**: используются в корневом Python-проекте и в CI (pip cache / install).

### Кандидаты на упрощение (можно объединить, чтобы не плодить дубли)
- **`docker-compose.ms-v2.yml` (корень)** vs **`backend/ms-v2/docker-compose.yml`**:
  - по сути оба поднимают `redis + api + worker` и оба монтируют `final_blyat_v3.csv` + `jobs` volume.
  - различия: имена сервисов (`ms-v2-*` vs `redis/api/worker`), пути env/volumes, и контекст сборки (в корневом compose — `context: ./backend/ms-v2`, во втором — `context: .` потому что он лежит внутри `backend/ms-v2`).
  - в `README.md` сейчас команда запуска ссылается на **`docker-compose.ms-v2.yml`** (строка “docker compose -f docker-compose.ms-v2.yml ...”).
  - решение: выбрать один compose как “главный”, второй удалить, и поправить `README.md`.

### Локальный мусор (не коммитится, но можно удалять с диска)
